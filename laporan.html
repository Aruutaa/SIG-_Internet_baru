Laporan Sistem

SDLC, user, database, dan analisis

Dokumentasi pengembangan WebGIS fasilitas kesehatan Kabupaten Purworejo.

1. Software Development Lifecycle

Model SDLC yang digunakan adalah Prototype. Model ini dipilih karena rancangan peta, filter, popup, radius, rute, layer OSM 3D Buildings, dan halaman developer perlu diuji dalam bentuk tampilan awal sebelum disempurnakan.

Perencanaan

Menentukan tujuan sistem, cakupan data, jenis user, kebutuhan peta, kebutuhan login, dan data fasilitas kesehatan yang akan dikelola.

Analisis

Mengidentifikasi kebutuhan titik fasilitas, batas administrasi, atribut layanan, status IGD, dokter, rawat jalan, rawat inap, konsultasi penyakit dalam, serta layer bangunan 3D dari OpenStreetMap.

Desain

Menyusun halaman beranda, peta, katalog data, login/buat akun, developer console, dan laporan. Struktur database dirancang menggunakan PostgreSQL/PostGIS.

Implementasi

Frontend menggunakan HTML, CSS, JavaScript, Leaflet, MarkerCluster, Routing Machine, Chart.js, dan OSMBuildings. Backend menggunakan PHP API. Database menggunakan PostgreSQL/PostGIS.

Pengujian

Menguji pencarian, filter, marker, popup, radius buffer, rute, fasilitas terdekat, ekspor CSV, login, buat akun, tambah data, edit data, hapus data, dan import SQL.

Pemeliharaan

Memperbarui data fasilitas, memvalidasi koordinat, menyesuaikan atribut layanan, menyimpan backup database, dan memperbaiki tampilan berdasarkan hasil evaluasi.

2. Tujuan sistem

Menginventarisasi fasilitas kesehatan, memvisualisasikan sebaran fasilitas, membantu pencarian fasilitas terdekat, menampilkan radius jangkauan, dan menyediakan pengelolaan data melalui developer console.

3. Kelas user

Pengunjung membuka peta, memakai filter, melihat popup, rute, radius, statistik, dan katalog data. Operator membantu input dan pembaruan data. Admin mengelola data, akun, dan validasi sistem.

4. Database kesehatan

Database memuat app_users, fasilitas_kesehatan, faskes_ketersediaan, wilayah_administrasi, jenis_fasilitas, layanan_kesehatan, fasilitas_layanan, kunjungan_bulanan, osm_3d_building_layer, dan activity_logs.

5. Analisis yang digunakan

Analisis meliputi sebaran fasilitas per jenis, sebaran per kecamatan, radius layanan 1 km sampai 5 km, fasilitas terdekat, rute menuju fasilitas, BPJS, IGD, rawat inap, ambulans, serta konsultasi dokter penyakit dalam.

6. Arsitektur sistem

Frontend menampilkan peta dan dashboard. Backend menghubungkan browser ke database. PostgreSQL/PostGIS menyimpan atribut dan geometri. GeoJSON dipakai sebagai fallback saat API belum aktif. OSMBuildings dipakai untuk visualisasi bangunan 3D.

Revisi database dan layer spasial

Layer bangunan menggunakan OpenStreetMap 3D Buildings. Database tetap menyediakan tabel cache OSM jika data bangunan perlu disimpan ke PostGIS.

OSM 3D Buildings

Bangunan ditampilkan dari OpenStreetMap Buildings pada halaman peta. Tampilan 3D menjadi konteks visual area sekitar fasilitas kesehatan.

Ketersediaan fasilitas

Atribut mencakup IGD dan jam operasional, dokter umum, dokter spesialis, dokter penyakit dalam, rawat jalan, rawat inap, serta konsultasi penyakit dalam.

Akses pengguna

Halaman login dilengkapi fitur buat akun. Developer Console dipakai untuk edit data fasilitas kesehatan secara terstruktur.

Kesesuaian instruksi

Sistem memuat SDLC, minimal dua jenis user, serta database geospasial PostgreSQL/PostGIS.

SDLC

Model Prototype dijelaskan dari tahap perencanaan sampai pemeliharaan.

User

Tersedia pengunjung, operator, dan admin. Admin/operator menggunakan login untuk mengelola data.

Database

Data atribut dan geometri disimpan dalam PostgreSQL/PostGIS serta dapat dibaca melalui API.
