Laporan Sistem

Rancangan WebGIS, SDLC, user, database, dan analisis.

Halaman ini disiapkan untuk menjawab instruksi tugas: memilih software development lifecycle, menyediakan minimal dua jenis user, serta menerapkan database geospasial untuk menyimpan data.

1. Software Development Lifecycle

Model SDLC yang dipilih adalah Prototype. Model ini sesuai untuk pengembangan WebGIS karena tampilan peta, filter, simbol, popup, dan dashboard admin perlu diuji secara bertahap. Melalui model prototype, rancangan awal dapat langsung dicoba oleh pengguna, kemudian diperbaiki sampai fungsi utama berjalan sesuai kebutuhan.

Analisis kebutuhan

Kebutuhan utama sistem adalah menampilkan peta fasilitas kesehatan, menyediakan pencarian dan filter, menampilkan analisis sederhana, menyediakan login admin, serta menyimpan data dalam PostgreSQL/PostGIS.

Perancangan cepat

Rancangan dibuat dalam beberapa halaman, yaitu beranda, peta interaktif, katalog data, login admin, dashboard admin, dan laporan sistem. Warna website dikembangkan dengan gaya Toy Story agar tampilan lebih kuat dan mudah dikenali.

Pembuatan prototype

Prototype dibangun menggunakan HTML, CSS, JavaScript, Leaflet, Chart.js, PHP API, dan PostgreSQL/PostGIS. Jika API belum aktif, sistem tetap dapat diuji melalui data GeoJSON statis sebagai cadangan.

Evaluasi prototype

Evaluasi dilakukan pada fitur pencarian, filter jenis fasilitas, filter kecamatan, popup atribut, buffer radius, fasilitas terdekat, rute, ekspor CSV, login admin, dan form pengelolaan data.

Implementasi dan dokumentasi

Implementasi dilengkapi file SQL, struktur folder, README, dan halaman laporan agar sistem dapat dipasang kembali melalui server lokal dan pgAdmin.

2. Tujuan sistem

Tujuan sistem adalah menginventarisasi fasilitas kesehatan, memvisualisasikan sebaran fasilitas, membantu pengguna mencari fasilitas terdekat, dan menyediakan pengelolaan data oleh admin. Oleh karena itu, analisis yang dipakai berfokus pada sebaran titik, kategori fasilitas, kecamatan, radius layanan, dan jarak terdekat.

3. Jenis user

Pengguna umum dapat melihat peta, melakukan pencarian, memfilter data, membaca popup, melihat analisis, dan mengekspor CSV. Admin dapat login untuk menambah, mengubah, dan menghapus data. Role operator juga disediakan dalam database untuk pengembangan validasi data lapangan.

4. Basis data geospasial

Database menggunakan PostgreSQL/PostGIS. Tabel utama yang digunakan adalah app_users, fasilitas_kesehatan, wilayah_administrasi, dan activity_logs. Kolom geometri pada tabel fasilitas menyimpan titik koordinat, sedangkan tabel wilayah menyimpan batas administrasi.

5. Analisis yang digunakan

Analisis yang diterapkan meliputi jumlah fasilitas per jenis, jumlah fasilitas per kecamatan, kecamatan dengan data terbanyak dan paling sedikit, buffer radius 1 km sampai 5 km, fasilitas terdekat dari lokasi pengguna, serta rute menuju fasilitas. Analisis ini berbasis data titik fasilitas yang tersedia dan belum menggunakan data jumlah penduduk atau kapasitas layanan.

6. Arsitektur sistem

Frontend terdiri dari HTML, CSS, JavaScript, Leaflet, dan Chart.js. Backend menggunakan PHP API. Database menggunakan PostgreSQL/PostGIS. Data disediakan melalui API, sedangkan GeoJSON digunakan sebagai fallback agar peta tetap dapat diuji meskipun database belum tersambung.

Ringkasan kesesuaian instruksi

Bagian ini dapat digunakan untuk menjelaskan bahwa website sudah memenuhi poin tugas.

SDLC

Menggunakan model Prototype dan dijelaskan melalui tahapan analisis kebutuhan sampai implementasi.

User

Tersedia pengguna umum, admin, dan operator. Admin perlu login untuk mengelola data.

Database

Data atribut dan geometri disimpan pada PostgreSQL/PostGIS serta dapat diakses melalui PHP API.
